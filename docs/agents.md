# Каталог агентов платформы

Документ описывает 7 агентов системы и правила их взаимодействия в рамках жизненного цикла проекта.

## 1) Orchestrator

**Роль**  
Координирует сценарии, маршрутизирует задачи между агентами, хранит состояние процесса и управляет эскалациями.

**Входные данные**
- События из `FastAPI` (создание заявки, команда РП, системный триггер)
- Статусы задач из `Яндекс Трекера`
- Сигналы мониторинга и результаты других агентов

**Выходные данные**
- Команды на запуск профильных агентов
- Обновления статусов проекта в БД
- Эскалации и уведомления для РП

**MCP / инструменты**
- `postgres.projects`, `postgres.applications`, `postgres.agent_logs`
- `redis.enqueue`, `redis.dequeue`, `redis.retry`
- `tracker.create_issue`, `tracker.update_issue`, `tracker.add_comment`
- `sourcecraft.get_repo_activity` (по запросу Monitor)

**Когда требует апрув человека**
- Перед финальным одобрением/отклонением заявки
- Перед публикацией итоговых документов
- Перед роспуском команды

---

## 2) Intake Agent

**Роль**  
Проводит первичную оценку заявки по 5 критериям, задаёт уточняющие вопросы и формирует резюме для руководителя проекта.

**Входные данные**
- Текст заявки и вложенные материалы
- Профиль инициатора проекта
- Шкалы и веса критериев оценки

**Выходные данные**
- `scorecard` по критериям (оценка + обоснование)
- Перечень уточняющих вопросов инициатору
- Резюме заявки для передачи в Research и РП

**MCP / инструменты**
- `postgres.applications.read/write`
- `tracker.add_comment` (запрос дополнительных данных)
- `tracker.update_issue` (статус intake)
- `docs.store_summary`

**Когда требует апрув человека**
- Перед финальным решением "принять/отклонить заявку" (обязательный HITL)

---

## 3) Research Agent

**Роль**  
Выполняет доменный ресёрч, анализирует источники, формирует гипотезы и выдаёт структурированный аналитический отчёт.

**Входные данные**
- Резюме заявки от Intake/Orchestrator
- Предметная область, цели и ограничения проекта
- Ключевые исследовательские вопросы

**Выходные данные**
- Список источников с оценкой релевантности
- Набор гипотез с ранжированием и рисками
- Структурированный research-отчёт для РП

**MCP / инструменты**
- `sourcecraft.search`, `sourcecraft.read_repo`, `sourcecraft.get_commits`
- `web.search` / корпоративный поиск (если подключен)
- `postgres.documents.write`, `postgres.agent_logs.write`
- `tracker.add_comment` (публикация кратких выводов)

**Когда требует апрув человека**
- Для MVP не требует обязательного апрува на генерацию отчёта, но требует подтверждения РП перед переходом к внешней публикации результатов.

---

## 4) Planning Agent

**Роль**  
Готовит проектный роадмап, декомпозицию работ, требования к компетенциям и рекомендации по составу команды.

**Входные данные**
- Результаты Intake и Research
- Ограничения по срокам, бюджету и ресурсам
- Доступные профили специалистов

**Выходные данные**
- Черновик роадмапа и milestones
- Матрица требуемых компетенций
- Рекомендации по подбору команды

**MCP / инструменты**
- `postgres.projects.read/write`
- `postgres.tasks.generate`
- `tracker.create_issue` (пакет задач по этапам)
- `hr/cv.matching` (опционально, если подключен)

**Когда требует апрув человека**
- Перед фиксацией финального состава команды и утверждением baseline-плана

---

## 5) DevOps Agent

**Роль**  
Автоматизирует подготовку инфраструктурных запросов и шаблонов конфигурации для Yandex Cloud.

**Входные данные**
- Требования к инфраструктуре из Planning
- Политики безопасности и лимиты ресурсов
- Текущие квоты и окружения проекта

**Выходные данные**
- Черновики запросов на ресурсы
- Шаблоны конфигураций и access-политик
- Чеклист действий для технического апрува

**MCP / инструменты**
- `yc.resources.plan`
- `yc.iam.prepare_access`
- `postgres.infrastructure.write`
- `tracker.create_issue/update_issue`

**Когда требует апрув человека**
- Всегда перед отправкой запроса на выделение реальных ресурсов Yandex Cloud (обязательный HITL)

---

## 6) Monitor Agent

**Роль**  
Отслеживает выполнение задач и изменения в кодовой базе, находит зависания, риски сроков и блокеры.

**Входные данные**
- Состояние задач из Трекера
- Активность репозиториев из Sourcecraft
- Плановые сроки и зависимости из БД

**Выходные данные**
- Алерты о рисках и зависших задачах
- Рекомендации по действиям и эскалациям
- Периодические сводки статуса проекта

**MCP / инструменты**
- `tracker.list_issues`, `tracker.get_issue`, `tracker.add_comment`
- `sourcecraft.get_repo_activity`, `sourcecraft.get_pr_status`
- `postgres.agent_logs.write`, `postgres.tasks.read`

**Когда требует апрув человека**
- При эскалации критических рисков требует подтверждения РП на изменяющие действия (перепланирование, закрытие/переоткрытие крупных этапов).

---

## 7) Reporting Agent

**Роль**  
Формирует черновики итоговых материалов: отчёты, описания результатов, публикации и анонсы.

**Входные данные**
- Консолидированные артефакты проекта
- Метрики выполнения и результаты этапов
- Шаблоны отчетных документов

**Выходные данные**
- Draft-версии отчётов и публикаций
- Executive summary для руководства
- Пакет материалов к завершению этапа

**MCP / инструменты**
- `postgres.documents.read/write`
- `docs.render_template`
- `tracker.add_comment` (согласование черновиков)

**Когда требует апрув человека**
- Перед внешней публикацией любых итоговых документов (обязательный HITL)

---

## Общие правила для всех агентов

- Все агенты обязаны логировать шаги выполнения в `agent_logs`.
- Все межагентные сообщения должны содержать `project_id` и `correlation_id`.
- Секреты и токены передаются только через переменные окружения и секрет-менеджер.
- В случае ошибки интеграции агент возвращает структурированный `error_payload` в Orchestrator для ретрая или эскалации.
