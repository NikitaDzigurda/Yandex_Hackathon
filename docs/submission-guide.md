# Как сдавать решения

## Общий процесс

### 1. Подготовка решения

Финальная версия репозитория команды должна быть оформлена так, чтобы агент-оценщик мог **воспроизводимо** развернуть и протестировать решение.

Обязательные элементы:
- `tasks/` — копии всех заданий хакатона (для справки)
- `docs/` — документация по решению выбранного задания (для оценки бизнес-критериев)
- `src/` — код решения выбранного задания (для оценки кода)
- `README.md` — обзор решения + **явно выбранное задание**
- `DEPLOY.md` — **только** воспроизводимые инструкции запуска/проверки/тестирования для агента-оценщика
- `docker-compose.yml` — решение **обязательно** упаковано в контейнеры через `docker compose`

### 2. Минимальная структура репозитория

```
team-repo/
├── README.md
├── DEPLOY.md
├── docker-compose.yml
├── tasks/
├── docs/
└── src/
```

### 3. Создание коммита

```bash
git add .
git commit -m "Финальная версия решения: [краткое описание]"
git push
```

### 4. Оценка агентом-оценщиком

Оценка **не происходит после каждого коммита**. Организаторы запускают оценку **разово** для финальной версии.

## Требования к репозиторию

### Что нужно сделать:
1. Изучите текст задания в `tasks/`
2. Реализуйте решение в `src/`
3. Опишите решение и бизнес-логику в `docs/`
4. Укажите выбранное задание и подготовьте базовое описание репозитория в корневом `README.md`
5. Подготовьте воспроизводимый запуск и тесты в `DEPLOY.md` (через `docker compose`)
6. Создайте финальный коммит

## Чеклист перед финальной сдачей
- **Выбранное задание** явно указано в корневом `README.md`
- **Документация** в `docs/` соответствует выбранному заданию и описывает, что именно реализовано
- **Код** находится в `src/`
- В корне есть `docker-compose.yml`
- Команды из `DEPLOY.md`:
  - собирают и запускают проект через `docker compose`
  - запускают тесты/проверки через `docker compose`
  - дают однозначный критерий «всё работает»
  - останавливают/чистят окружение
