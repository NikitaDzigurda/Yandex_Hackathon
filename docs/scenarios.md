# Сценарии автоматизации (MVP)

Документ описывает два сквозных сценария: путь заявки и запуск глубокого ресёрча.  
Формат каждого шага: кто действует, что происходит технически, какой статус фиксируется в БД, где требуется HITL.

---

## Сценарий 1: Путь заявки (от подачи до задачи в Яндекс Трекере)

### Шаг 1. Подача заявки
- **Кто действует:** человек (инициатор проекта)
- **Что происходит технически:** инициатор отправляет заявку через API (`POST /applications`) с описанием идеи и контактными данными.
- **Статус в БД:** в `applications.status` фиксируется `submitted`, в `projects.status` создаётся/обновляется запись со статусом `intake_pending`.
- **HITL:** нет

### Шаг 2. Регистрация события и запуск intake
- **Кто действует:** Orchestrator
- **Что происходит технически:** Orchestrator получает событие о новой заявке, создаёт `correlation_id`, пишет запись в `agent_logs` со статусом `pending`, ставит задачу Intake Agent в очередь.
- **Статус в БД:** `applications.status = scoring`; в `agent_logs.status = pending`.
- **HITL:** нет

### Шаг 3. Оценка заявки по 5 критериям
- **Кто действует:** Intake Agent
- **Что происходит технически:** агент получает заявку, оценивает её по 5 критериям, формирует `scorecard` и при необходимости список уточняющих вопросов.
- **Статус в БД:** `applications.scorecard` заполнен (JSONB), `agent_logs.status = success`, `projects.status = intake_in_progress`.
- **HITL:** нет

### Шаг 4. Уточнение данных у инициатора (если нужно)
- **Кто действует:** агент + человек (инициатор)
- **Что происходит технически:** Intake Agent публикует вопросы в связанной задаче/комментарии и ожидает ответы; после ответов пересчитывает оценку и summary.
- **Статус в БД:** временно `applications.status = scoring`; после повторной оценки обновляются `scorecard` и `summary`.
- **HITL:** нет (операционная коммуникация)

### Шаг 5. Подготовка резюме для решения РП
- **Кто действует:** Intake Agent -> Orchestrator
- **Что происходит технически:** агент передаёт в Orchestrator итоговое резюме заявки, баллы, риски и рекомендацию (`approve`/`reject`).
- **Статус в БД:** `projects.status = awaiting_pm_decision`; в `agent_logs` фиксируется завершение этапа intake.
- **HITL:** **да (обязательный)** — решение принимает РП

### Шаг 6. Финальное решение по заявке
- **Кто действует:** человек (РП)
- **Что происходит технически:** РП подтверждает решение через API/интерфейс: одобрить или отклонить заявку.
- **Статус в БД:**  
  - при одобрении: `applications.status = approved`, `projects.status = approved_for_research`;  
  - при отклонении: `applications.status = rejected`, `projects.status = closed_rejected`.
- **HITL:** **да (обязательный)**

### Шаг 7. Создание задачи в Яндекс Трекере
- **Кто действует:** Orchestrator
- **Что происходит технически:** при статусе `approved` Orchestrator вызывает API Трекера, создаёт задачу проекта, добавляет краткое резюме и метаданные заявки.
- **Статус в БД:** в `tasks` создаётся запись с `tracker_issue_id`; `projects.status = tracker_issue_created`.
- **HITL:** нет

### Шаг 8. Подтверждение запуска следующего этапа
- **Кто действует:** Orchestrator
- **Что происходит технически:** отправляется команда на запуск сценария Research Agent на основе резюме Intake.
- **Статус в БД:** `projects.status = research_pending`.
- **HITL:** нет

## Сценарий 2: Глубокий ресёрч (от резюме до структурированного отчёта)

### Шаг 1. Инициация ресёрча
- **Кто действует:** Orchestrator
- **Что происходит технически:** Orchestrator формирует research-job из резюме заявки, целей проекта и контекста домена, создаёт запись в `agent_logs`.
- **Статус в БД:** `projects.status = research_in_progress`; `agent_logs.status = pending`.
- **HITL:** нет

### Шаг 2. Сбор источников
- **Кто действует:** Research Agent
- **Что происходит технически:** агент выполняет поиск по подключенным источникам (включая Sourcecraft и доступные внешние/внутренние базы), собирает candidate-источники.
- **Статус в БД:** промежуточная запись в `documents` типа `research_sources_draft`; в `agent_logs` добавляется действие `collect_sources`.
- **HITL:** нет

### Шаг 3. Анализ и фильтрация релевантности
- **Кто действует:** Research Agent
- **Что происходит технически:** агент ранжирует источники, отбрасывает нерелевантные, фиксирует обоснования и ограничения по качеству данных.
- **Статус в БД:** обновляется черновик документа источников; `agent_logs` получает `action = rank_sources`, `status = success`.
- **HITL:** нет

### Шаг 4. Формирование гипотез
- **Кто действует:** Research Agent
- **Что происходит технически:** на основе отобранных источников агент генерирует набор гипотез, оценивает риски и ожидаемую ценность, присваивает приоритеты.
- **Статус в БД:** в `documents` создаётся/обновляется `doc_type = hypotheses_draft`.
- **HITL:** нет

### Шаг 5. Сбор структурированного отчёта
- **Кто действует:** Research Agent
- **Что происходит технически:** агент собирает финальный отчёт с разделами: контекст, источники, гипотезы, риски, рекомендации, следующие шаги.
- **Статус в БД:** в `documents` сохраняется `doc_type = research_report`, увеличивается `version`; `agent_logs.status = success`.
- **HITL:** нет

### Шаг 6. Передача отчёта в Orchestrator и Трекер
- **Кто действует:** Orchestrator
- **Что происходит технически:** Orchestrator сохраняет ссылку/тело отчёта, добавляет summary в комментарий к задаче Трекера и назначает follow-up задачи.
- **Статус в БД:** `projects.status = research_completed`; в `tasks.status` обновляются связанные задачи.
- **HITL:** нет

### Шаг 7. Решение о внешней публикации
- **Кто действует:** человек (РП)
- **Что происходит технически:** РП принимает решение, можно ли публиковать/распространять результаты ресёрча вне рабочей группы.
- **Статус в БД:** при ожидании решения `projects.status = awaiting_publication_approval`; после решения обновляется соответствующий статус.
- **HITL:** **да (обязательный)** перед внешней публикацией
