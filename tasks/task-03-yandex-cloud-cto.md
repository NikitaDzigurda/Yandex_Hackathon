# Задача хакатона: Платформа автоматизации управления проектной деятельностью на базе ИИ-агентов

## Контекст и проблематика

Центр технологий для Общества ведёт проекты на стыке медицины и ИИ (далее — другие направления: экология, наука, образование, культура). Жизненный цикл проекта включает множество этапов: от приёма заявок и первичной оценки до ресерча, формирования команды, выделения ресурсов в Yandex Cloud, реализации и публикации результатов. Сейчас многие операции выполняются вручную, что создаёт узкие места:

- **Субъективная и ручная** первичная оценка заявок по 5 критериям
- **Долгий и трудозатратный** глубокий ресерч с привлечением внешних источников (интернет, научные статьи)
- **Долгое согласование** документов в Трекере (соглашения, договоры, акты)
- **Ручной поиск** ресурсов под проект, описание требований, отбор кандидатов
- **Ручное нарезание** инфраструктуры и выдача доступов команде
- **Лавина коммуникаций** (в т.ч. в Telegram, Трекер, почта), сложно ничего не пропустить и отслеживать прогресс

Требуется **система автоматизации управления проектной деятельностью при помощи ИИ-агентов**, которая покрывает весь цикл и снижает ручную нагрузку, сохраняя контроль и качество решений.

---

## Цель задачи

Разработать **концепцию и/или прототип платформы** для комплексной автоматизации управления проектной деятельностью с использованием ИИ-агентов. Платформа должна поддерживать:

1. Автоматизацию этапов жизненного цикла проекта (от заявки до публикации и масштабирования).
2. Взаимодействие с существующими инструментами: **Яндекс Трекер**, репозитории в **Sourcecraft**, сервисы **Yandex Cloud**.
3. Работу в связке с людьми: РП, инициаторы проектов, студенты партнёрских вузов и ШАД.

Результат хакатона — не обязательно готовый продукт; допустимы: архитектура платформы, описание агентов и сценариев, MVP по одному–нескольким этапам, демонстрация ключевых сценариев.

## Рекомендации технологического стека

Ниже — ориентиры по технологиям. Стек не обязателен: команды вправе выбирать другие инструменты, если это обосновано и не противоречит правилам хакатона.

| Назначение | Рекомендации | Зачем |
|------------|--------------|--------|
| **Облако и инфраструктура** | Yandex Cloud (Compute Cloud, Managed Service, Object Storage, серверы и контейнеры) | Соответствие экосистеме Центра, возможность интеграции с существующими ресурсами и будущим внедрением. |
| **ИИ и LLM** | YandexGPT (API в Yandex Cloud), другие LLM по выбору (OpenAI-совместимые API, открытые модели) | Оценка заявок, ресерч, генерация документов, диалоги с инициаторами; предпочтительно использование моделей, доступных в РФ. |
| **Оркестрация агентов** | LangChain, LlamaIndex, собственный координатор на Python/Node.js, очереди (Redis, RabbitMQ, Yandex Message Queue) | Связка шагов сценариев, вызов инструментов (поиск, API), асинхронность и устойчивость к сбоям. |
| **Бэкенд и API** | Python (FastAPI, Flask), Node.js; при необходимости — сервисы в Yandex Cloud (Serverless, Cloud Functions) | Быстрая разработка прототипа, удобная интеграция с LLM и внешними API. |
| **Интеграции** | API Яндекс Трекера, API Sourcecraft (Git), Telegram Bot API, Yandex Cloud API (IAM, ресурсы) | Реальная связка с текущими инструментами Центра; в прототипе достаточно одной–двух интеграций с демонстрацией остальных в описании. |
| **Фронтенд / интерфейс** | React, Vue, простой дашборд (Streamlit, Gradio) или Telegram-бот как основной интерфейс | Удобство для РП и команды; на хакатоне допустим минимальный, но понятный UI. |
| **Хранение данных** | PostgreSQL, SQLite, Yandex Managed Service for PostgreSQL; кэш — Redis при необходимости | Хранение заявок, состояний сценариев, результатов агентов, логов. |
| **Развёртывание** | Docker, docker-compose; опционально — Yandex Cloud (контейнеры, serverless) | Лёгкий запуск решения жюри и воспроизводимость окружения. |

**Примечание.** Если организаторы предоставляют тестовые доступы к Трекеру, Sourcecraft или песочнице Yandex Cloud — имеет смысл заложить их использование в прототип; иначе достаточно описать интеграции и показать демо на моках.

## Этапы жизненного цикла проекта (для автоматизации)

| № | Этап | Что делается сейчас | Возможная роль ИИ-агентов |
|---|------|---------------------|---------------------------|
| 1 | Приём и первичная оценка заявок | Оценка по 5 критериям вручную, при необходимости — досбор информации от инициатора | Оценка заявок по критериям, формулирование запросов на уточнение, ведение диалога с инициатором |
| 2 | Глубокий ресерч | Ручной поиск и анализ источников (интернет, статьи), формулирование гипотез | Поиск и первичный анализ источников, выдвижение и ранжирование гипотез, подготовка структурированного отчёта |
| 3 | Планирование и поиск ресурсов | Роадмэп, описание задачи и гипотез, требования к компетенциям и инфраструктуре, поиск команды | Генерация базовых документов, роадмэпа, описания компетенций, помощь в подборе и ранжировании кандидатов |
| 4 | Выделение инфраструктуры и доступов | Ручное нарезание ресурсов в Yandex Cloud, выдача доступов команде | Автоматизация запросов на ресурсы, шаблоны конфигураций, выдача и отзыв доступов по ролям/этапам |
| 5 | Реализация проекта | Отслеживание коммуникаций, прогресса, изменений в репозиториях; помощь РП ничего не пропустить | Мониторинг репозиториев и трекера, сводки по прогрессу, напоминания, выявление рисков и «зависших» задач |
| 6 | Подготовка к завершению | Ручная подготовка материалов для публикации (что сделано, результаты) | Черновики отчётов, описаний результатов, структура публикаций |
| 7 | Завершение и масштабирование | Анонсы, публикации, роспуск команды; решение о новой серии или масштабировании | Подготовка анонсов, сбор метрик, рекомендации по масштабированию или новой итерации проекта |

## Критерии оценки проектов (Центр)

Их можно заложить в логику агентов первичной оценки и приоритизации:

- **Значимость** — полезность для общества, понятность широкой аудитории  
- **Технологичность** — возможность реализации с помощью технологий Яндекса (Yandex Cloud)  
- **Научность** — научная база, новизна, наличие научного партнёра  
- **Практичность** — ясные сроки и цели (до 1 года, желательно до 6 мес.), определённый образ результата и его применимость  
- **Масштабируемость** — возможность масштабирования в другие области, регионы, страны

## Оценивание на хакатоне

### Критерии оценки решения

- Охват этапов жизненного цикла проекта и устойчивая работа ИИ-агентов (выполнение действий, обработка ошибок и граничных случаев).
- Архитектура агентов и сценариев, оркестрация и возможность воспроизводимо развернуть решение (docker-compose или понятная инструкция).
- Удобство интерфейса для РП и команды: наглядные статусы, понятные действия, доступные сводки результатов агентов.
- Связка с экосистемой Центра (Яндекс Трекер, Sourcecraft, Yandex Cloud, Telegram) и осмысленность вывода ИИ с учётом критериев оценки проектов Центра там, где это уместно.
- Оригинальность сценариев автоматизации и собственный вклад команды сверх формулировок задания.
- Качество презентации и демонстрации ценности решения на защите.

## Ожидаемые результаты от участников

- **Описание платформы**: архитектура, роли ИИ-агентов, интеграции с Трекером, Телеграм, Sourcecraft, Yandex Cloud.

## Ограничения и требования

- Учёт существующих инструментов: **Telegram**, **Яндекс Трекер**, **Sourcecraft**, **Yandex Cloud** (желательно использование в прототипе или явное описание интеграций).
- Команда проекта: РП, ментор, студенты партнёрских вузов и ШАД — агенты должны дополнять, а не подменять их роли без явного согласования.
- Безопасность и согласованность: автоматизация документов (договоры, акты) и доступов должна предусматривать контроль и аудит.

## Уточняющие вопросы к организаторам / заказчикам

Ниже — зафиксированные ответы заказчика на уточняющие вопросы. Они помогают участникам спроектировать платформу и прототип под реальные процессы Центра.

### 1. Приоритет этапов

**Вопрос.** Какой этап (или 2–3 этапа) наиболее критичен для автоматизации в первую очередь? Имеет ли смысл ограничить хакатон одним–двумя этапами для более глубокой проработки?

**Ответ.** Все этапы жизненного цикла остаются в поле зрения; участники могут выбрать любые из них для автоматизации. Предпочтительнее глубоко проработать отдельные этапы (столько, сколько успеют за время хакатона), чем поверхностно охватить все сразу. Ключевой ориентир — чтобы представленное решение можно было затем применять на практике.

---

### 2. Допустимые технологии

**Вопрос.** Есть ли жёсткие требования к стеку (например, только определённые LLM, только Yandex Cloud)? Разрешены ли внешние API (поиск, научные базы)?

**Ответ.** Жёстких требований к стеку нет. Главное — чтобы решение работало и приносило ценность.

---

### 3. Формат заявок и данных

**Вопрос.** Есть ли структурированный формат заявок (JSON, форма в Трекере и т.п.)? Можно ли использовать анонимизированные реальные заявки для демо?

**Ответ.** Заявка обычно приходит как плоский неструктурированный текст в одном поле. Это поле попадает в описание задачи в Яндекс Трекере; его нужно извлекать оттуда. Если командам понадобятся примеры таких заявок, организаторы их предоставят.

---

### 4. Интеграции

**Вопрос.** Планируется ли предоставление тестовых инстансов Трекера, репозиториев в Sourcecraft, песочниц Yandex Cloud для участников?

**Ответ.** У всех участников на время хакатона есть бесплатный доступ к Yandex Cloud и Sourcecraft. Яндекс Трекер тоже нужно использовать: интеграции желательно показывать «вживую». Допустимы моки и заглушки, но они оцениваются ниже, чем реальные работающие интеграции.

---

### 5. Роль человека

**Вопрос.** Где решения должны оставаться за человеком (например, финальный вердикт по заявке, утверждение документов), а где допустима полная автоматизация?

**Ответ.** На этапе внедрения и тестирования предпочтительно оставлять за человеком подтверждение действий. После успешного тестирования отдельные шаги можно постепенно переводить на полную автоматизацию.

---

### 6. Многопроектность

**Вопрос.** Нужна ли поддержка множества параллельных проектов с разными стадиями и разными командами в одной платформе?

**Ответ.** Да, такую поддержку закладывать нужно.
