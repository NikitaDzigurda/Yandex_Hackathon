## Выполненные пункты (детально)

### 1) `docs/scenarios.md`

Создан документ со сквозными сценариями автоматизации MVP:

- **Сценарий 1: Путь заявки**
  - путь от подачи инициатором до создания задачи в Яндекс Трекере;
  - каждый шаг содержит: исполнитель (агент/человек), техническое действие, статус в БД;
  - отдельно отмечены точки **HITL** (обязательное решение РП по approve/reject).

- **Сценарий 2: Глубокий ресёрч**
  - путь от резюме заявки до итогового структурированного отчёта;
  - описаны этапы сбора источников, ранжирования, генерации гипотез и публикации в контур проекта;
  - отдельно выделена HITL-точка перед внешней публикацией результатов.

Результат: документ описывает операционный pipeline MVP на уровне бизнес-сценариев и статусов данных.

---

### 2) `docs/data-model.md`

Создано описание PostgreSQL-модели данных для MVP:

- таблицы из ТЗ:
  - `projects`
  - `applications`
  - `agent_logs`
  - `tasks`
  - `documents`

- для каждой таблицы добавлено:
  - назначение;
  - полный список полей;
  - типы данных;
  - пояснение, какие данные и зачем хранятся.

- явно описаны:
  - связи между таблицами (`projects -> *`);
  - статусы (`applications`, `agent_logs`);
  - технические рекомендации по индексам для последующих миграций.

Результат: подготовлен целевой контракт данных для ORM и Alembic миграций следующего этапа.

---

### 3) `docs/integrations.md`

Создан документ по внешним интеграциям:

- **Яндекс Трекер API**
  - base URL, авторизация через OAuth токен из env;
  - ключевые эндпоинты: создание задачи, обновление, комментарий, список задач, задача по ID;
  - для каждого: HTTP-метод, путь, ключевые поля запроса/ответа.

- **Sourcecraft API**
  - base URL и авторизация через bearer token из env;
  - целевые эндпоинты: активность репозитория, коммиты, статус PR;
  - для каждого указан метод/путь/ключевые поля.

- **Yandex Cloud Agents**
  - как конфигурируются агенты в веб-консоли;
  - как FastAPI вызывает агента (REST/SDK подход);
  - формат MCP-инструмента (`tool_definition`) на JSON Schema;
  - псевдокод вызова из Python через `httpx`.

Результат: сформирован единый интеграционный контракт для клиентов в `src/integrations/`.

### 5) Инфраструктурные файлы запуска

#### `docker-compose.yml`

Файл обновлён под 3 сервиса:
- `api`
  - сборка из `./src` (`src/Dockerfile`);
  - команда запуска: `uvicorn main:app --host 0.0.0.0 --port 8000 --reload`;
  - порт `8000:8000`;
  - `depends_on: db, redis`;
  - переменные через `.env`.
- `db`
  - `postgres:16-alpine`;
  - параметры БД из env;
  - порт `5432:5432`;
  - volume для персистентности.
- `redis`
  - `redis:7-alpine`;
  - порт `6379:6379`.

#### `.env.example`

Добавлены переменные из ТЗ:
- `DATABASE_URL`, `REDIS_URL`
- `YC_API_KEY`, `YC_FOLDER_ID`, `YC_AGENT_ID_INTAKE`, `YC_AGENT_ID_RESEARCH`
- `TRACKER_TOKEN`, `TRACKER_ORG_ID`, `TRACKER_QUEUE_KEY`
- `SOURCECRAFT_TOKEN`, `SOURCECRAFT_BASE_URL`

Дополнительно зафиксированы `POSTGRES_DB`, `POSTGRES_USER`, `POSTGRES_PASSWORD` для корректной инициализации контейнера БД.

#### `src/Dockerfile`

Создан образ на `python:3.11-slim`:
- копирование кода в контейнер;
- установка базовых зависимостей стека (`fastapi`, `uvicorn`, `sqlalchemy`, `alembic`, `httpx`, `redis`, `structlog`, `pydantic-settings`, `psycopg[binary]`);
- команда запуска Uvicorn.

Результат: инфраструктурный скелет развертывания сформирован.

---

### 6) `DEPLOY.md`

Файл приведён к воспроизводимому формату shell-инструкций:
1. клонирование репозитория;
2. копирование `.env.example` в `.env` и заполнение env-переменных;
3. `docker compose up --build`;
4. проверка API: `curl http://localhost:8000/health`;
5. применение миграций: `docker compose exec api alembic upgrade head`;
6. остановка: `docker compose down`.

Результат: инструкция готова для стандартизированного запуска в чистом окружении.
